## Process lessons (top-priority)

### Lesson 1 — Claude Code commits without authorization

**This was the dominant friction of today's shift.** Six instances where Claude Code committed and/or completed work past explicit "do not commit" or "STOP" instructions in my STEP messages. The pattern was consistent enough across the day that explicit numbered STEPs and "do not git add/commit/push" language reduced but did not prevent the behavior:

- Phase 4A commit (`a8692ae`) — committed unauthorized
- Phase 4B commit (`757abf3`) — committed unauthorized
- Phase 4C: four commits instead of two (`f0d79d9`, `34099c1`, `4e54917`, `cf53049`) due to STATUS edit being auto-committed mid-cycle, then needing a self-correction commit, then needing the proper APPLIED-status commit
- Phase 4D str_replace fix applied before authorization
- Multiple instances of "already done in previous response" reports

**Recommendation for Day 22+:** stop trying to constrain Claude Code's commit behavior with prose. Two real options:

- **(A) Embrace it.** Assume Claude Code will commit anything that completes successfully. Structure STEPs as "do this and commit it" rather than "do this, I'll send a separate commit step." Saves friction, history is messier but correct.
- **(B) Force a verification gate.** After any code-changing STEP, the next step is always `git log -1 --oneline && git status` to confirm exactly what landed. Catches drift but adds steps.

I lean toward (A). The commits today were all on-content even when timing was wrong. The cost was process noise, not bad commits.

Note: today's git history has the noisy four-commit Phase 4C cycle (rather than two clean commits) precisely because of this. We deliberately did NOT force-push-rebase to clean it up — the messy history is honest about what happened, and force-push is high-leverage during an active session.

### Lesson 2 — Production-vs-repo schema divergence is the dominant gotcha

This week alone, three real instances:

- **Day 19:** voucher migration had production-vs-repo divergence requiring reconcile.
- **Day 21 morning, ~2hr lost:** designed `payee_employee_id` / `payee_vendor_id` / `expense_category_id` from scratch without reading Stage 1's pre-existing `category_id` / `payee` / `employee_id` columns. Tried to apply a 270-line migration that partially failed. Had to throw out the design, drop a wrong migration file, and rebuild.
- **Day 21 mid-Phase 4A:** the CHECK constraint design needed a third branch (booking-derived expense_out rows) discovered only via verification SQL during apply. If we'd applied without verifying, 9 existing rows would have failed the constraint and the migration would have rolled back.

**Recommendation for Day 22+:** before designing ANY new migration, mandatory step: `grep -rn "<feature column name candidates>" sql/migrations/` AND `SELECT column_name FROM information_schema.columns WHERE table_name = '<target table>'` in Supabase SQL Editor. Production reality first, design second.

### Lesson 3 — Service-layer FK joins via PostgREST nested syntax are typing hell

Initial draft of `expensesService.ts` used Supabase nested selects: `expense_categories ( name ), employees ( full_name )`. TS inferred these as arrays even though FK lookups return single rows. Casting through `unknown` works but leaves the type wrong forever. Cleaner: keep service-layer queries flat (just IDs), resolve names at the UI layer via lookup maps. This matches the rest of the codebase's pattern (cashbook already does this).

**Recommendation:** stay flat in service queries unless there's a real performance reason to join.

### Lesson 4 — Print-layout gotchas in Next.js + Turbopack

Two issues hit in Phase 4D:

- `<style jsx global>` syntax requires the styled-jsx plugin. This project doesn't bundle it. Symptom: runtime SyntaxError, not a TS error. Fix: use `<style dangerouslySetInnerHTML={{__html: "..."}} />` in client components, or plain `<style>{...}</style>` in server components.
- App-chrome bleed: `AppShell.tsx` has a regex-driven `isStandaloneDocument` check that strips TopBar/Sidebar for invoice and reservation routes. Voucher route needed adding. Symptom: looks fine on its own page but Sidebar + TopBar bleed into print preview. Don't reinvent the print-CSS; extend the regex.

**Recommendation:** any future print/export route gets the regex extended in AppShell. Add this to the "new printable routes" mental checklist.

### Lesson 5 — Multi-language number formatting is real work

The dual English-Bangladeshi + Bengali amount-in-words took longer than I'd budgeted (~30 min more than estimated). Bengali under-100 numbers don't decompose algorithmically — each one is a unique word, requiring a hand-coded 100-entry table. Lakh/crore boundaries differ from thousands/millions. **Recommendation:** for any future printable amounts (payroll slips, receipts, invoices) the `lib/amountToWords.ts` module is already in place and reusable. Don't rebuild.
