## What Shipped in #58a (the big one)

**Schema additions:**
- `refunds.pre_adjusted BOOLEAN NOT NULL DEFAULT FALSE` — TRUE for
  auto-created refunds where `paid_amount` was already decremented at
  checkout time. `disburse_refund` branches on this flag to UPDATE the
  existing payment row (no double-decrement) instead of inserting a new
  negative payment.
- `payments.refund_id UUID NULL REFERENCES refunds(id) ON DELETE SET NULL`
  — explicit FK linking auto-created adjustment payments to their refund
  row. Replaces fragile `notes LIKE` matching. Indexable. `ON DELETE SET
  NULL` preserves payment rows if refund is ever deleted.

**`checkout_booking` — new step 3.5:**
Inserted between rooms-to-cleaning (step 3) and `update_booking_total`
(step 4). Reads `booking_rooms.nights` after the CTE has already reduced
them, computes `effective_total = rooms_total + extra_charge - discount`,
and if `paid_amount > effective_total`, inserts a pending refund row +
negative payment row for the difference. Ordering is critical: negative
payment INSERT fires `trg_sync_paid_amount` (decrements `paid_amount`)
BEFORE `update_booking_total` reduces `total_amount` — so
`chk_paid_not_exceed_total` is satisfied at every point.

**`checkout_booking_room` — same overpayment detection block:**
Inserted before the inline `UPDATE bookings SET total_amount` when
`p_deduction_amount > 0`. Reads current financials, computes new
effective total, inserts refund + negative payment if overpaid, THEN
proceeds with the total reduction.

**`disburse_refund` — pre_adjusted branch:**
`pre_adjusted = TRUE` → finds the existing payment row by `refund_id` FK
and UPDATEs its `method` to the actual disbursement method. No new INSERT.
No second `paid_amount` decrement.
`pre_adjusted = FALSE` → existing INSERT behaviour unchanged (normal
cancellation-path pending refunds).

**UI validation fix (`BookingsClient.tsx` line 2102):**
Discount validation was checking `amt > remainingAfterPayment`. In
overpayment scenarios `remainingAfterPayment` is negative (e.g. -2000),
so an empty discount field (`amt = 0`) evaluated `0 > -2000 = true` and
raised "Discount cannot exceed remaining balance of ৳-2,000", silently
blocking checkout submission. Fix: `amt > 0 && amt > remainingAfterPayment`
— validation only fires when a discount is actually entered.

**SECURITY DEFINER on 3 RPCs:**
Applied via `ALTER FUNCTION ... SECURITY DEFINER` on live DB and reflected
in migration file + `sql/schema/07-functions.sql`. Details in the RLS
finding below.

## Design Decision Documented

**Arif:** "Always create refund + deny-as-tip pattern."

- System auto-creates a pending refund for any overpayment at checkout.
  No operator decision required during checkout flow itself.
- Operator handles via Timeline modal afterward:
  - **Disburse** (cash/bKash/etc.) → `mark_disbursed`
  - **Guest gifted / tip** → `deny_refund` with reason
- "Tips are not hotel revenue — denied refunds are the tip accounting
  record."
- This means every overpayment is tracked, even when no money changes
  hands. Denied refunds serve as tip receipts.

## Significant Architectural Finding — RLS Doc-Drift

**Symptom:** `disburse_refund` called via UI updated `refunds.status` and
`refunds.disbursement_method` correctly, but `payments.method` stayed
`'other'`. Same function called directly in SQL Editor worked correctly.

**Root cause:** `disburse_refund` was `SECURITY INVOKER` (Postgres default
— no `SECURITY DEFINER` declared). SQL Editor runs as `service_role`
(bypasses RLS). UI runs as `authenticated` via anon key. The `UPDATE
payments SET method = ...` inside the pre_adjusted branch ran as the
`authenticated` role.

`sql/schema/06-rls-policies.sql` declares:
```sql
CREATE POLICY "Authenticated can update payments"
  ON public.payments FOR UPDATE TO authenticated USING (true);
```
But the **live DB does not have this policy**. RLS is enabled on payments,
and without the UPDATE policy, authenticated users cannot UPDATE payment
rows. The INSERT policy exists (so normal payment recording works), but
UPDATE does not — explaining why the `disburse_refund` pre_adjusted UPDATE
silently affected 0 rows with no error raised.

**Fix applied:** `SECURITY DEFINER` on `disburse_refund`,
`checkout_booking`, and `checkout_booking_room`. Functions now run with
owner privileges → RLS bypassed for all table operations inside → behavior
is consistent regardless of calling role.

**Remaining issue:** The underlying policy doc-drift (06-rls-policies.sql
out of sync with live DB) is logged as #19. The SECURITY DEFINER approach
is the right long-term pattern for sensitive RPCs anyway — it's Supabase's
standard recommendation.
