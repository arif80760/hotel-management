## Architectural Decision Pending — #58

**Scenario:** Guest pays in full at check-in. Leaves a day early.
- bookings.paid_amount = total
- checkout_booking RPC reduces booking_rooms.nights
- update_booking_total reduces bookings.total_amount
- Now paid_amount > total_amount
- chk_paid_not_exceed_total constraint fires → checkout HARD FAILS

**Same architectural class affects:** discount on fully-paid booking 
(BK-1063 yesterday — modal blocked overpayment-creating discount).

**Three options for Day 12 to decide:**

A. **Auto-create refund row.** checkout_booking RPC inserts a 
   negative payment row for the overpayment difference. The trigger 
   (now updated in #50) re-aggregates paid_amount. Constraint holds. 
   Operator processes actual refund through existing disburse_refund 
   workflow.

B. **Block checkout, force refund-first workflow.** Operator must 
   process refund through separate flow before checkout. UI directs.

C. **Relax constraint.** Allow paid > total. Bad — silent overpayments
   without tracking.

**Recommended: Option A.** Matches Phase 8.5 disbursement architecture
already in place. Refund disbursements are already supported via 
disburse_refund RPC. Estimated 4-8 hours of careful work.
