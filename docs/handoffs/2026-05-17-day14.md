## Phase A — Auth Model (decided earlier, recorded here)

- Two roles: admin and staff.
- Both roles do everything currently built; identical permissions today.
- The admin-only boundary exists for future features (accounts, payroll) and
  user management.
- Login: email + password only. Admin creates all accounts; no public sign-up.
- Roles live in the profiles table, keyed to auth.users.id.

## Security — Before and After

BEFORE (start of day): the live database had anon-role RLS policies granting
full CRUD — including DELETE — on bookings, payments, guests, rooms,
booking_guests, plus anon SELECT on 8 tables. The anon key ships in the
frontend. Anyone could read, modify, or delete all hotel data without
authenticating.

AFTER (end of day): zero anon policies on any table (verified —
SELECT COUNT(*) ... WHERE 'anon' = ANY(roles) returns 0). Every database
operation requires an authenticated Supabase session. Route protection is
enforced server-side by middleware. The anon key can perform no database
operation. #40 fully closed; #19 closed.

## Significant Findings During the Day

- Session storage split: a server-side auth check on the document pages was
  prototyped in B1a, then reverted, because pre-D1 sessions lived in
  localStorage (server could not see them). D1 fixed the root cause; D3
  restored the check.
- middleware.ts is a new file — Next.js needs a dev-server restart to pick it
  up; hot reload is not enough. (A blank /login page during D2 testing was
  caused by stale dev-server state, resolved by restarting the server.)
- API routes: middleware excludes /api/. The one current API route
  (/api/employees/provision) authenticates itself via Bearer token. Any
  FUTURE API route must implement its own auth — middleware does not cover
  /api/ by design.
