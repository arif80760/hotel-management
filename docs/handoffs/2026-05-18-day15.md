## Accounts Architecture — Designed and Committed (9fd58f7)

Full spec at docs/architecture/accounts.md. Key decisions:

- A mid-range bookkeeping tool for the hotel team — NOT professional
  accounting software. Mental model: a cash register / daybook.
- Four money buckets: Cash in Hand, Bank, bKash, Nagad. Each has a running
  balance. The admin can only SPEND from Cash in Hand (real hotel policy —
  admin has no bank access; the software enforces it).
- Six transaction types: revenue_in, expense_out, transfer, injection,
  loan_received, loan_repayment. Distinct types keep revenue/expense reports
  honest (an injection and a loan are not income; a loan repayment is not an
  expense).
- Booking payments map to buckets automatically by payment method
  (cash->Cash, bank->Bank, bkash->bKash, nagad->Nagad). This integration seam
  is the riskiest part — if wrong, Cash in Hand drifts from reality.
- Expenses: dynamic categories (create + autocomplete), always funded from
  Cash, each generates a professional payment voucher (sequential number,
  payee, amount in figures and words, no approval workflow).
- Payroll: salary/advance/bonus recorded as Salary-tagged expense
  transactions linked to an employee. No hour tracking or calculation.
- Loans: tracked as loan records with partial-repayment support.
- Day-close: manual action; closing balance becomes next day's opening;
  missed days are closed sequentially oldest-first.
- Receipt images and guest documents stored in Supabase Storage. Decision
  made to move to Supabase Pro ($25/mo) — 100 GB storage comfortably covers
  both. Image compression on upload to be built in.
- Accounts is ADMIN-ONLY — gated by the Phase E current_user_role() pattern.
- Inventory: a SEPARATE later feature. The expense table reserves a
  product-purchase hook-point so inventory plugs in later without a rebuild.
