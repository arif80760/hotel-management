## What Stage 1 Built

Database foundation for the Accounts feature (spec: docs/architecture/
accounts.md). Migration: sql/migrations/2026-05-18-accounts-core-stage1.sql
(filename dated 05-18; actually run on 05-19).

- account_transaction_type enum — 6 values: revenue_in, expense_out,
  transfer, injection, loan_received, loan_repayment.
- accounts table — the four money buckets, seeded with FIXED hardcoded UUIDs:
    Cash in Hand : a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11  (is_spendable = true)
    Bank         : b0eebc99-9c0b-4ef8-bb6d-6bb9bd380a22
    bKash        : c0eebc99-9c0b-4ef8-bb6d-6bb9bd380a33
    Nagad        : d0eebc99-9c0b-4ef8-bb6d-6bb9bd380a44
  Later stages reference buckets by these stable UUIDs, not by name lookup.
  No balance column — balances are computed by summing account_transactions.
- account_transactions table — every money movement. Amount always positive;
  direction given by from_account_id / to_account_id. chk_txn_accounts CHECK
  constraint (boolean-OR form, fails closed) enforces valid from/to per type.
  Reserved nullable columns for later stages: category_id, voucher_number,
  payee, employee_id, booking_payment_id, loan_id, receipt_image_url. FKs
  declared where the referenced table exists (employees, payments); FKs for
  category_id and loan_id deferred to the stages that create those tables.
- 5 indexes on account_transactions (txn_date, from_account_id,
  to_account_id, type, booking_payment_id partial).
- Admin-only RLS on both tables via the Phase E current_user_role() pattern.
  accounts is SELECT-only (no write policies — the four bucket rows are fixed).

Verified: migration ran clean; 4 buckets seeded with correct UUIDs;
COUNT = 4; 5 RLS policies; 5 indexes; chk_txn_accounts correctly REJECTED a

## Key Design Decisions

- Balances are COMPUTED (sum of transactions), never stored — no drift.
- Booking-payment method -> bucket mapping (decided, for the future
  integration stage): cash->Cash in Hand; bank_transfer/card/online/other
  ->Bank; bkash->bKash; nagad->Nagad. (Architecture doc Section 3.1 still
  shows only the original 4-method mapping — update it when the integration
  stage is built.)
- One row per transaction (Option A): a transfer is a single row with both
  from and to set; not double-entry.
- accounts table has no write policies — buckets are permanent fixtures.
