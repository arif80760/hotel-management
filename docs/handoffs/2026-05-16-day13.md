## What Shipped — #58b + #58c

**The bug:** A discount applied to a fully-paid booking was written by a
post-RPC Step 2 UPDATE in checkoutNormal/checkoutWithOverride. Lowering
the effective total pushed paid_amount above it; chk_paid_not_exceed_total
fired at the PostgREST layer. checkout_booking's step 3.5 overpayment
detection never saw the discount because it arrived after the RPC committed.

**The fix (Option B1 — compute, refund, then write):**
- checkout_booking gains three params: p_additional_discount_amount
  (NUMERIC), p_additional_discount_reason (TEXT), p_additional_discount_by
  (UUID).
- Step 3.5 sources the discount from the parameter, not the bookings
  column, so overpayment detection is correct.
- New step 3.6 writes the additional_discount_* columns inside the RPC,
  after step 3.5's negative-payment row has already decremented
  paid_amount — so the constraint holds at every point.
- Discount write removed from checkoutNormal and checkoutWithOverride
  Step 2 payloads; the discount is now passed to the checkout_booking
  RPC call instead.

**UI fix (#58b):** validateAndBuildDiscount capped the discount at
remainingAfterPayment, which is 0 for a fully-paid booking — blocking any
discount with "cannot exceed remaining balance of taka 0". Ceiling changed
to the billable total (totalAmount + extra charge - early deduction). Error
message corrected.

**#58c:** Checkout modal now shows an informational "Pending refund of
taka X will be created" hint (sky-blue) when a discount produces an
overpayment. Informational only — does not block submission.

## Runtime Type Mismatch — Found and Fixed

First test on BK-1035 failed at runtime: "column additional_discount_by
is of type uuid but expression is of type text". The RPC declared
p_additional_discount_by as TEXT; the bookings column is UUID. PostgREST
implicitly coerces text->uuid at the HTTP layer (so the old Step 2 write
worked), but PL/pgSQL does not coerce on assignment inside a function
body. Fixed via a follow-up migration changing the parameter to UUID
(DROP + CREATE, PostgREST schema reload). No TypeScript change needed.

## Known Behavior — Documented, Not a Bug

bookings.total_amount stores the gross room total (e.g. taka 8,000), NOT
the discount-adjusted figure. The discount is tracked separately in
additional_discount_amount. Effective amount due = total_amount -
additional_discount_amount. This is intended (confirmed with Arif). Note:
a raw query of total_amount vs paid_amount can look like a balance is
owed when it is not — readers must also account for
additional_discount_amount. The invoice already displays this correctly.
