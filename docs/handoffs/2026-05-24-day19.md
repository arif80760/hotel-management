## In production

### Voucher number sequence reconciled (`805881a`, applied 2026-05-24)

The voucher number migration drafted Day 18 (commit `ffa9b7c`) was queued for Day 19 application. When we ran it, verification revealed production already had a **different, simpler** `next_voucher_number()` function — single never-resetting sequence `public.voucher_seq` instead of the per-year `voucher_seq_{YYYY}` design in the repo file. Origin of the prior version unknown; never tracked in this repo.

Reconciliation: dropped the old function and `voucher_seq`, applied the repo's per-year design as canonical. Side effect: `voucher_seq_2026` ended at `last_value=4` after four verification calls during reconcile. Left as-is per the migration's "gaps are acceptable" design — first real expense voucher will be `EV-2026-0005`.

This was the first concrete instance this project of "production diverges from repo at unknown point." Migration file STATUS updated, divergence documented in the file footer.

### Booking-payment integration UPDATE branch fix (`11f304c`, applied 2026-05-24)

`fn_sync_account_transactions` UPDATE branch was rewriting daybook rows with `NEW.created_at::date` and `NEW.created_at` — the original pre-adjustment payment's timestamp, which can be weeks or months old. When designing the day-close immutability trigger, we realized this would cause `disburse_refund` to fail silently with an opaque error if the pre-adjustment's `created_at` fell on or before any close_date.

Fix: UPDATE branch now writes `txn_date = current_date` and `created_at = now()`, reflecting the disbursement moment. INSERT branch unchanged. The trigger's own design comment ("at this moment money has physically moved, so the daybook row is born") was correct about intent; only the implementation drifted.

Discovered while drafting day-close. Bundled into Day 19 because shipping day-close without the fix would have left a latent footgun.

### Day-close mechanism (`f4d7c0f`, applied 2026-05-24)

`day_closes` table created in production. Schema per `docs/architecture/accounts.md` §14: `(id UUID, close_date DATE UNIQUE, opening_balance NUMERIC(12,2), closing_balance NUMERIC(12,2), closed_by UUID, closed_at TIMESTAMPTZ)`. RLS enabled with SELECT + INSERT policies for `authenticated`. No UPDATE or DELETE policies — closes are immutable by design.

Immutability trigger `trg_check_account_transactions_immutability` registered on `account_transactions`. BEFORE INSERT/UPDATE/DELETE FOR EACH ROW. Blocks writes where `txn_date <= MAX(day_closes.close_date)`. Allows all writes if `day_closes` is empty (pre-bootstrap; never the case in production after this migration).

Bootstrap seed inserted inline at apply time: `2026-05-23` with both opening and closing = ৳423,550.02 (Cash in Hand as of end of May 23, computed via SUM CASE WHEN `to_account_id` = Cash bucket THEN amount ELSE `-amount`).

Smoke tests confirmed:
- INSERT with `txn_date = 2026-05-23` rejected with the expected RAISE (P0001).
- INSERT with `current_date` succeeded.

Migration file: `sql/migrations/2026-05-24-day-close.sql` (244 lines).

### Day-close service layer (`f943d22`)

`services/dayCloseService.ts` (263 lines). Three exports:

- `getDayCloseStatus()` returns `{ lastClosedDate, missedDays, canCloseToday, todaysOpeningBalance }`. Reads latest day_closes row, computes backlog between lastClosed+1 and today-1.
- `getMissedDays()` — convenience wrapper for the UI banner.
- `closeDay(closeDate)` — refuses non-today dates (Path B scope), checks auth, refuses if backlog exists, computes today's Cash in Hand net delta, inserts the row with `closed_by` from the auth session.

Catch-up flow (closing a backlog day with its own review screen per §11.3) deferred. Bootstrap UI deferred (handled via migration-time seed).

Known caveat documented in-file: `todayIso()` uses JS Date in local zone; DB trigger uses Postgres `current_date`. On a UTC+6 server these align except at the UTC midnight boundary. Future refactor should route "what is today" through Postgres.

### Day-close UI card (`720a1c8`)

Close Day card added to `app/accounts/cashbook/CashbookClient.tsx`, sits above the success banner.

Three render states:
- **Already closed today** — green "Closed" pill, no button.
- **Backlog present** — amber banner listing missed dates, button disabled.
- **Otherwise** — opening balance, today's Cash in Hand activity list with signed amounts, computed closing preview, "Close Today" button.

`loadDayCloseStatus()` wired into the existing mount effect as fire-and-forget — non-fatal failure leaves the card hidden, rest of page usable. Imports `ACCOUNT_IDS` from `accountsService` for Cash in Hand bucket filtering.

Smoke-tested end-to-end via real browser click — May 24 close landed at ৳423,550.02 (no Cash in Hand activity that day), state updated in-place via re-fetch.

### Soft delete on account_transactions (`56f7cf0`, applied 2026-05-24, committed 2026-05-29)

`deleted_at TIMESTAMPTZ` column added to `account_transactions` (nullable, no default). Partial index `idx_acct_txn_not_deleted` on `(id) WHERE deleted_at IS NULL` for fast live-row queries. `account_balances` view replaced with same shape but with `AND t.deleted_at IS NULL` added to the LEFT JOIN — soft-deleted rows no longer contribute to bucket balances.

Service-layer changes in the same commit:
- `deleteTransaction()` now does `UPDATE ... SET deleted_at = now()` instead of `DELETE`. Idempotent guard via `.is("deleted_at", null)` on the WHERE clause.
- `getTransactions()` adds `.is("deleted_at", null)` filter so soft-deleted rows are invisible to the cashbook UI.

Design decisions (locked Day 19):
- Soft delete applies only to manual rows (booking_payment_id guard unchanged in `deleteTransaction`).
- Closed-day immutability applies because soft delete is an UPDATE — trigger rejects soft-delete attempts on closed-day rows. UI does NOT yet disable trash icon on closed-day rows; that's a Day 20+ polish item.
- Soft-deleted rows hidden by default everywhere. No "show deleted" toggle. Restorability is a manual SQL operation if ever needed.

Migration was applied to production during the Day 19 shift but the file and service edits were never committed before the shift ended (Eid break). Cleaned up on Day 20 (May 29) — STATUS updated to APPLIED, committed and pushed as `56f7cf0`.

## Process lessons

**1. Trust the verbatim rule for chat output, not just file content.** Process lesson #3 from Day 18 was about Claude Code summarizing file contents. Day 19 found a sibling failure mode: I (Claude) produced text that looked paste-ready but contained annotations meant as commentary (arrows like `← OLD value`, placeholders like `YOUR_TOKEN_HERE`). When the user copied that into Supabase or curl, the result was a syntax error or an authenticated-as-the-string-`YOUR_TOKEN_HERE` request. Cost: ~30 min on one auth misdiagnosis (we wrongly concluded a real PAT was invalid based on a curl test that ran against the placeholder string). Lesson: when sending pasteable content in chat, the canonical source is the file on disk, not the chat block. Annotations belong outside the code block, not inside.

**2. SSH beats PAT for git auth on this machine.** First push of the shift failed with `Device not configured` (no stored credentials in Claude Code's shell). User's own terminal then failed with `Password authentication is not supported`. PAT generation went sideways twice — first token had wrong scope, second token's diagnostic curl was misread. Total time spent on auth: ~45 min. SSH setup (key + GitHub upload + config + remote URL switch + test) took ~10 min and worked first try. PAT remains useful for one-off CI tokens; for ongoing dev work on this Mac, SSH is the answer. Switch is permanent for this repo.

**3. Discovering a precondition mid-shift is a normal cost, not a planning failure.** Day-close design discussion exposed the booking-payment UPDATE-branch bug (back-dated `txn_date` would fail the immutability trigger). Without the amendment, day-close ships with a latent footgun; with it, day-close ships cleanly. The amendment wasn't in the original plan but was the right thing to do that shift. Bundling it cost ~1.5 hours but avoided shipping a known-broken state. Process lesson: when a precondition surfaces during design review, don't defer it to "save time" — the deferral cost is higher than the immediate cost.

**4. Apply → verify is one tight loop, not two streams.** Multiple times this shift we ran verification queries before the apply step had actually succeeded (or had errored silently). Each round of "but Q1 still shows the old function!" cost ~10 min to diagnose. Lesson: after an apply step, the only legitimate next message is "Success" + verification, in that order, with verification gated on the success message. No parallel work threads, no running ahead.

**5. Production divergence from repo is a real failure mode, not a curiosity.** The voucher migration's prior version had reached production via an unknown path — no commit in this repo created it. If we hadn't run the verification queries and noticed `voucher_seq` (the wrong sequence name) in `pg_sequences`, we'd have shipped a broken year-rollover and a confused codebase. The class of bug warrants vigilance going forward: every migration apply should be followed by a verification that production matches the file just applied (`pg_get_functiondef`, `pg_get_viewdef`, column-existence checks). The voucher reconcile + the soft-delete post-Eid verification both followed this pattern; future migrations should too.

**6. The Day 18 pacing warning is real.** Day 18 said: "Arif tends to push for 'more features, longer shifts.' If you're 4+ hours into a session and starting to feel friction, trust that signal." Day 19 hit that threshold at ~6h 30min (after voucher reconcile, trigger amendment, push auth resolution, day-close end-to-end) and was asked to add three more features (audit trail, soft delete, expense UI) for the final hours. Initial pushback was warranted but my time math was wrong — I undercounted available time by ~2.5h. User was right to flag. Lesson: when pushing back on scope, double-check the time math before leading with the pushback. Trust the user's clock over my mental accounting in a long thread.
